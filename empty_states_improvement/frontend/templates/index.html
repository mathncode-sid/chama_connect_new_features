Accessibility Features

Making components accessible to all users

🎯 Keyboard Navigation

  • Tab - Navigate between buttons
  • Enter - Activate button
  • Space - Activate button
  • Esc - Close modal

🗣️ Screen Reader Support

  • All buttons have descriptive aria-labels
  • Empty states have semantic heading hierarchy
  • Loading states announce their purpose
  • Modals trap focus for screen readers

👁️ Visual Accessibility

  • Color contrast ratio: WCAG AA (4.5:1)
  • Focus indicators: Visible on all interactive elements
  • Icons paired with text labels
  • Animations respect prefers-reduced-motion

When to Use Skeleton Loaders

✓ Good Use Case

When you know content is loading immediately (connection is good)

Example: Dashboard cards loading

✗ Wrong Use Case

When there's no data and you're not certain when data will appear

Example: Empty search results (use empty states instead)

⚡ Best Practice

Show skeleton for known data, empty state for missing data

Example: Show skeleton for profile pic, empty state if none uploaded

💡 Design Principles

State Clarity

Users should always know if the app is loading, empty, or has an error.

Action-Oriented

Every empty state should suggest next steps (CTA buttons).

Reduce Cognitive Load

Simple messaging, clear hierarchy, helpful hints.

Distinguish States

Empty ≠ Loading ≠ Error. Use different visual treatment.
